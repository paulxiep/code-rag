# Cutting a code-rag-mcp release

Releases are built by the manually-triggered [release.yml](../.github/workflows/release.yml) GitHub Action. You bump the version, push, click "Run workflow", and the Action builds the single `code-rag-mcp` binary for Linux/macOS/Windows on GitHub-hosted runners and creates a draft release with the zips attached. You smoke-test, then publish.

## Per-release checklist

1. **Bump version** in [crates/code-rag-mcp/Cargo.toml](../crates/code-rag-mcp/Cargo.toml) ŌĆö `version = "..."`. The `code-raptor` crate version doesn't ship in the release zip (it's only used as an internal lib by `code-rag-mcp`'s `ingest` subcommand), but bump it too if the orchestration changed materially so dev/harness work stays auditable.

2. **Update [development_log.md](../development_log.md)** with a new entry (or extend the most recent if same milestone).

3. **Commit + push** to `main` (or your default branch).

4. **Trigger the release workflow**:
   - Go to Actions ŌåÆ "Release" on the left ŌåÆ "Run workflow".
   - Type the version (e.g. `v0.1.0`) ŌĆö must match what you bumped to in step 1.
   - Leave "Create as draft" checked.
   - Click "Run workflow".

5. **Wait** for the build (~3-5 min for the matrix). When it finishes, a draft release appears at https://github.com/paulxiep/code-rag/releases.

6. **Smoke-test** on a fresh dir or scratch machine:
   - Download the zip for your platform from the draft release.
   - Extract somewhere persistent. Verify `code-rag-mcp.config.yaml` is alongside the exe.
   - Edit `code-rag-mcp.config.yaml`: set `target_path` to a non-portfolio test repo (and `workspace: true` if it's a parent folder of sub-projects).
   - Double-click `code-rag-mcp` (or run with no args from a terminal). Verify it writes `.claude/skills/code-rag.md`, `.mcp.json`, and `.gitignore` entry into the target dir, then exits.
   - (For Claude Code to find the binary) put `code-rag-mcp` on PATH or hand-edit the generated `.mcp.json` to use the absolute path.
   - Open Claude Code in the test repo. Ask "what does this project do?" ŌĆö verify the agent runs `code_rag_reindex mode=full` for the first ingest and answers via `code_rag_overview`.

7. **Publish** the draft release. Done.

## What ships in each zip

- `code-rag-mcp(.exe)` ŌĆö single binary. Serves MCP, runs setup, runs ingest. Subsumes the standalone `code-raptor` CLI for end users.
- `code-rag-mcp.config.yaml` ŌĆö setup config template. User edits `target_path` + optionally `workspace: true`, then runs the exe.
- `LICENSE` ŌĆö Apache-2.0 (skipped if absent at build time).
- `README.txt` ŌĆö install + first-run notes (generated by the workflow).

The reranker ONNX model (~88 MB) is **not** bundled. The binary fetches it from HuggingFace on first run; offline users set `CODE_RAG_RERANKER_DIR` or pass `--no-rerank`.

The Claude Code skill is **not** a separate file in the zip ŌĆö it's embedded into the binary at compile time via `include_str!`, and the exe writes it into the user's target dir during setup. Skill source of truth: [crates/code-rag-mcp/skills/code-rag.md](../crates/code-rag-mcp/skills/code-rag.md). Updates to that file flow into the next release automatically.

## Targets shipped today

The release matrix covers:
- `x86_64-unknown-linux-gnu`
- `aarch64-apple-darwin` (Mac M-series)
- `x86_64-pc-windows-msvc`

Add Intel Mac (`x86_64-apple-darwin`) or other targets to the `matrix.include` list in [release.yml](../.github/workflows/release.yml) when there's user demand.
